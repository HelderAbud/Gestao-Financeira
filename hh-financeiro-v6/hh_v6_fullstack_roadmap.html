Pilar 1 — Backend Spring Boot
API REST + Autenticação JWT + PostgreSQL

Pilar 3 — Sincronização na nuvem
Dados acessíveis em qualquer dispositivo, em tempo real
Com backend próprio, a sincronização acontece naturalmente a cada request. Para atualizações em tempo real (ex: dois dispositivos abertos), optamos por polling ou WebSocket conforme a necessidade.
Escolha da estratégia de sincronização:
✅ Recomendado — API própria (Spring Boot)
Controle total dos dados. Sem lock-in de vendor. Escalável. Dados criptografados em trânsito (HTTPS) e em repouso. Funciona com qualquer banco. Custo previsível.
Firebase Realtime DB
Mais rápido para implementar. Mas: lock-in Google, preço pode escalar mal, menos controle de segurança, dados em servidor de terceiro.

Decisões técnicas importantes — por que essa stack
☕
Por que Spring Boot e não Node.js?
Spring Boot tem Spring Security maduro com suporte nativo a JWT, JPA para ORM robusto e tipagem forte com Java/Kotlin. Para dados financeiros, a segurança built-in e o ecossistema corporativo são vantagens reais. Se preferir Node, NestJS seria o equivalente mais próximo.
⚛️
Por que Next.js e não Vite + React puro?
Next.js App Router oferece SSR para a tela de login (melhor SEO e segurança), deploy trivial na Vercel, e file-based routing que estrutura o projeto naturalmente. Para um app com autenticação, o middleware de rota do Next.js é perfeito para proteger páginas.
🐘
Por que PostgreSQL e não MySQL ou MongoDB?
PostgreSQL tem suporte nativo a JSON (útil para campos flexíveis), transações ACID robustas, suporte a JSONB para dados semiestruturados, e é o banco padrão do Supabase. Para dados financeiros relacionais, SQL é a escolha correta — MongoDB seria superdimensionado e arriscado.
🔐
Segurança na migração da v5 para v6
A senha do usuário nunca viaja para o backend em texto puro. O cliente faz hash local (SHA-256) antes de enviar. O backend aplica bcrypt ao hash recebido antes de armazenar. Dados da v5 podem ser migrados via importação de backup .hhbak pelo próprio usuário após criar conta.
Por onde começar — próximos passos concretos
Sequência recomendada de execução
1 Criar repositório no GitHub com estrutura frontend/ e backend/
2 Inicializar Spring Boot via start.spring.io com dependências: Web, Security, JPA, PostgreSQL, JWT
3 Criar conta no Supabase e configurar banco PostgreSQL + copiar connection string
4 Implementar entidades + auth JWT + endpoint POST /auth/login funcionando localmente
5 Inicializar Next.js com npx create-next-app@latest + Tailwind + estrutura de pastas
6 Criar tela de login consumindo a API · configurar React Query · proteger rotas com middleware Next.js
7 Deploy: push para GitHub → conectar Vercel ao repo → conectar Render ao repo backend → testar em produção
8 Escrever README com badges, screenshots, diagrama de arquitetura e instruções de setup
